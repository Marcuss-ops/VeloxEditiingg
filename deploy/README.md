# deploy/ — operator-facing deployment layout

This directory holds the Ansible-driven deployment manifest for the Velox
master. Files here are committed; the actual **secrets** live in
`group_vars/vault.yml`, which the operator generates and encrypts with
`ansible-vault` (the encrypted form never lands in our repo unless the
operator explicitly chooses to commit it, which is also safe).

## Layout

```
deploy/
├── README.md                                  ← this file
├── velox-server.env.example                   ← canonical master env template (PR B)
├── validate-master-env.sh                     ← master env validator (PR B)
├── inventory/
│   └── production.ini.example                 ← Ansible inventory template (PR C)
├── group_vars/
│   ├── all.yml                                ← non-secret vars, committed (PR C)
│   └── vault.yml.example                      ← placeholder for vault (PR C)
├── templates/
│   └── velox-server.env.j2                    ← Jinja2 env template (PR C)
└── playbooks/
    ├── bootstrap-ssh.yml                      ← operator user + sshd hardening (PR C)
    ├── deploy-master-config.yml               ← render env + restart master (PR C)
    └── rollback.yml                           ← restore last backup (PR C)
```

## Mandatory env vars before first deploy

Before running any playbook, ensure `/etc/velox-server.env` contains these
critical variables (not just the defaults from the template):

- **`MASTER_PUBLIC_URL`** — the URL clients and workers use to reach the
  master. Must be a real, reachable hostname or public IP, NOT the placeholder
  `velox-master.example.com`. Workers use this to call back; the admin UI uses
  this for external access. If unset, workers and clients cannot discover the
  master.

- **`VELOX_ADMIN_TOKEN`** — the admin API token. Must NOT be the placeholder
  `CHANGE_ME_set_admin_token_here`. Generated via `openssl rand -hex 32`.

## First-time setup

1. **Inventory**: copy and edit.
   ```bash
   cp deploy/inventory/production.ini.example deploy/inventory/production.ini
   $EDITOR deploy/inventory/production.ini
   ```
   Replace every `CHANGE_ME` with the real IP/hostname. Add
   `deploy/inventory/production.ini` to your `.gitignore` so the real
   addresses don't leak into the repo.

2. **Vault**: create and encrypt.
   ```bash
   cp deploy/group_vars/vault.yml.example deploy/group_vars/vault.yml
   $EDITOR deploy/group_vars/vault.yml
   ansible-vault encrypt deploy/group_vars/vault.yml
   ```
   Add `deploy/group_vars/vault.yml` to your `.gitignore` to prevent
   accidental commits of the unencrypted form. The encrypted form is safe
   to commit; CI decrypts at runtime via `ANSIBLE_VAULT_PASSWORD`.

3. **Bootstrap SSH** — run once per new host to set up the operator user
   and harden sshd:
   ```bash
   ansible-playbook -i deploy/inventory/production.ini \
     --ask-vault-pass \
     deploy/playbooks/bootstrap-ssh.yml
   ```

4. **Deploy configuration** — render the master env from the Jinja2
   template, validate it, and restart the service:
   ```bash
   ansible-playbook -i deploy/inventory/production.ini \
     --ask-vault-pass \
     deploy/playbooks/deploy-master-config.yml
   ```

5. **Rollback** — restore the most recent env backup if a deploy went bad:
   ```bash
   ansible-playbook -i deploy/inventory/production.ini \
     deploy/playbooks/rollback.yml
   ```

## Backups before deploy

Backups are NOT generated by these playbooks. They live in
`/var/backups/velox/<ISO-8601 timestamp>/velox-server.env` and are produced
by your CI workflow (or a pre-deploy script) before invoking
`deploy-master-config.yml`. `rollback.yml` picks the most recent of these
by mtime.

Minimal pre-deploy backup script (run as root on the master):

```bash
ts="$(date -u +%Y-%m-%dT%H-%M-%SZ)"
mkdir -p "/var/backups/velox/$ts"
install -o root -g velox -m 0640 /etc/velox-server.env \
  "/var/backups/velox/$ts/velox-server.env"
```

## Vault password distribution

* **Operator laptops** — store the password in `~/.vault-velox` (mode 0600)
  and pass `--vault-password-file ~/.vault-velox`. Never inline in shell
  history.
* **GitHub Actions** — store the password as a repository-level Secret
  named `ANSIBLE_VAULT_PASSWORD`. Reference it in the workflow as
  `env: ANSIBLE_VAULT_PASSWORD: ${{ secrets.ANSIBLE_VAULT_PASSWORD }}`.
* **Other CI** — use whatever secret-store mechanism your platform
  provides (GitLab CI Masked Variables, Jenkins Credentials Binding, etc.).

## Local development / rendering preview

To render the master env locally without running ansible full-stack:

```bash
pip install jinja2-cli
jinja2 deploy/templates/velox-server.env.j2 \
       deploy/group_vars/all.yml \
       --format=yaml \
       --extra "vault_velox_admin_token=$(openssl rand -hex 32)"
```

Combine this with `deploy/validate-master-env.sh` to confirm the rendered
env would pass CI without committing it anywhere.

## Safety summary

* `deploy-master-config.yml` validates the env **before** any systemd
  restart. If the env is invalid, the play fails and the service is
  untouched.
* `rollback.yml` validates the restored env before restarting.
* `bootstrap-ssh.yml` reloads sshd only after `sshd -t` confirms the new
  config parses.
* All env files are written root-owned, group-velox, mode 0640.
