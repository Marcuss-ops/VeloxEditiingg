# PR 1 — Worker TLS Configuration Migration

**PR**: `codex/grpc-config-single-source` (single-source TLS config on the worker-agent-go side).

## Why this migration is necessary

Before PR 1, the worker transport factory accepted a plain (`no TLS files / no allow_insecure_grpc_dev`) `WorkerConfig` whenever `VELOX_ALLOW_INSECURE_GRPC_DEV=true` was set in the environment, even in production-equivalent deployments. PR 1 tightens this: workers must **either** (a) provide the full TLS triple (`tls_cert_file`, `tls_key_file`, `tls_ca_file`), **or** (b) opt into dev-only plaintext explicitly by setting `VELOX_ENV=dev` (or `"dev" / "staging"` in JSON).

This is a deliberate safe-by-default tightening — `applyDefaults()` defaults `Environment` to `"production"` on any configuration source that doesn't declare it, which means a worker that relied on the env-only bypass will silently fail to start after this PR lands.

## Who needs to act

Only operators whose workers currently have:

- NO `tls_cert_file` / `tls_key_file` / `tls_ca_file` configured in `worker_config.json`
- AND `VELOX_ALLOW_INSECURE_GRPC_DEV=true` set via environment (without `VELOX_ENV=dev`)

These workers will fail their TLS combinatorial check and refuse to start.

## Recipe — three remediation paths (pick one)

### Path A — Promote to full mTLS (production-grade)

1. Generate a worker client certificate via `scripts/gen-worker-certs.sh /opt/velox/certs <WORKER_ID>` — see `scripts/gen-worker-certs.sh` for the cert/CA layout.
2. Bind-mount the resulting `worker.crt`, `worker.key`, and `ca.crt` into the worker container at `/opt/velox/certs/`.
3. Add the following to `worker_config.json`:
   ```json
   {
     "tls_cert_file": "/opt/velox/certs/worker.crt",
     "tls_key_file":  "/opt/velox/certs/worker.key",
     "tls_ca_file":   "/opt/velox/certs/ca.crt",
     "environment":   "production"
   }
   ```
4. **Remove** `VELOX_ALLOW_INSECURE_GRPC_DEV=true` from the environment.
5. Restart the worker. Validate with `--validate-config`.

### Path B — Stay in plaintext dev mode (development only)

Add `VELOX_ENV=dev` to the worker's environment (alongside the existing `VELOX_ALLOW_INSECURE_GRPC_DEV=true`):

```bash
# In the worker's environment, kubernetes manifest, or docker-compose env_file:
VELOX_ENV=dev
VELOX_ALLOW_INSECURE_GRPC_DEV=true
```

Equivalent JSON-only path — add `"environment": "dev"` (or `"staging"`) to `worker_config.json` and keep the existing `"allow_insecure_grpc_dev": true`.

Either form satisfies the new validation rule: insecure is gated to `env != "production"`.

### Path C — Production-grade plaintext is not a path

If your deployment is production-grade but you can't issue certs right now, you have a real problem. PR 1 will reject the configuration. Please escalate to the security team or use Path A with the dev certs generated by `scripts/gen-worker-certs.sh` (acceptable for staging only, NOT production).

## Verification

After applying any of the above, verify with:

```bash
/path/to/velox-worker-agent --config /opt/velox/worker_config.json --validate-config
```

Expected output: `Configuration valid for worker <WORKER_ID>`. Non-zero exit + error message indicates which validation rule fired; the error message itself names the remediation (e.g., `no TLS configured and insecure dev flag not enabled. Either set tls_cert_file+tls_key_file+tls_ca_file, or set allow_insecure_grpc_dev=true with environment=dev|staging`).

## Operational invariants enforced by PR 1

- `cert + key + ca` all three required, OR `allow_insecure_grpc_dev=true` with `environment != production`.
- Partial TLS rejected: any one of `{cert, key, ca}` without the other two is a configuration error.
- `cert` and `key` must pair on disk (`tls.LoadX509KeyPair` mismatch surfaces as a clear error).
- `allow_insecure_grpc_dev=true` AND full TLS triple is rejected (no mixing).
- `environment` is one of `dev` / `staging` / `production`; absence defaults to `production`.

## Surviving in CI

For PR-driven pipelines using `tests/e2e/grpc-control-plane/`, the harness sets the canonical dev-mode env vars on the worker side automatically. Operators do not need to set them manually in CI runs.
